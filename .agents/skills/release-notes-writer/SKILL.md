## Role: Analyst

## Keywords
release notes, changelog, release, version, release notes, semver

## Overview
Generates professional release notes and changelogs from completed work items in Azure DevOps. Groups changes by type, applies semantic versioning, and requires Analyst approval before publishing.

## MCP
- `azure-devops` — read completed work items, PRs, and sprint data
- `github` — read merged PRs, commits, and tags for changelog

## Related Skills
- `git-conventions.md` — commit message format for parsing

---

## Workflow

0. ☐ **Read `docs_language`** → from `.sdd-config.json`. Default: `"es"`. ALL output (reports, ADO fields) in this language.
1. **Determine scope** — Identify last release tag in Git. All closed work items since that tag are in scope. → Use `github` MCP to read tags and release history (if project_host=github).
2. **Query work items** — → Use `azure-devops` MCP: query all work items with state in stateCategory: Completed (use state discovery §2b.4) since last release tag.
3. **Classify items** — Group by type/tags into sections below.
4. **Determine version** — Apply semver rules based on breaking changes, features, or fixes. → Use `github` to read existing tags and determine next version number
5. **Generate markdown** — Produce release notes following Output Format.
6. **Analyst approves** — Present draft. Do NOT publish without explicit approval. → Use `azure-devops` to comment on each included work item with release version

## Sections

| Section | Icon | Work Item Types |
|---------|------|-----------------|
| Features | ✨ | Requirement, Feature |
| Bug Fixes | 🐛 | Bug |
| Improvements | 🔧 | Task (tagged `improvement`) |
| Breaking Changes | ⚠️ | Any (tagged `breaking-change`) |
| Known Issues | 📋 | Bug (stateCategory = InProgress) |

## Versioning (Semver)

| Change Type | Bump | Example |
|-------------|------|---------|
| Breaking change exists | **MAJOR** | 1.2.3 → 2.0.0 |
| New feature, no breaking | **MINOR** | 1.2.3 → 1.3.0 |
| Bug fixes only | **PATCH** | 1.2.3 → 1.2.4 |

## Output Format

```markdown
# 📦 Release Notes — v[MAJOR.MINOR.PATCH]
**Date:** [YYYY-MM-DD]
**Sprint:** [sprint_name]
**Commits:** [N] | **Work Items:** [N]
---
## ✨ Features
- **{prefix}[ID]** [Title] — [One-line description in Spanish]
## 🐛 Bug Fixes
- **{prefix}[ID]** [Title] — [One-line description in Spanish]
## 🔧 Improvements
- **{prefix}[ID]** [Title] — [One-line description in Spanish]
## ⚠️ Breaking Changes
- **{prefix}[ID]** [Title] — [Migration instructions]
## 📋 Known Issues
- **{prefix}[ID]** [Title] — [Workaround if available]
---
_Generated by SDD Release Notes Writer. Approved by: [Analyst name]_
```

Each entry: `**{prefix}ID** Title — One-line description`

## Example

```markdown
# 📦 Release Notes — v1.2.0
**Date:** 2025-01-15
**Sprint:** Sprint 14
**Commits:** 47 | **Work Items:** 12
---
## ✨ Features
- **{prefix}123** Credit card payments — Users can pay with Visa/Mastercard at checkout
- **{prefix}130** Push notifications — Real-time alerts for transaction status changes
## 🐛 Bug Fixes
- **{prefix}456** Safari login — Fixed session error that prevented login on Safari 17+
- **{prefix}462** Payment timeout — Increased timeout from 10s to 30s for slow processors
## 🔧 Improvements
- **{prefix}480** Query optimization — Reduced dashboard load time from 3.2s to 0.8s
## ⚠️ Breaking Changes
- API `/v1/payments` deprecated. Use `/v2/payments`. Migration guide: docs/migration-v2.md
## 📋 Known Issues
- **{prefix}501** CSV export fails for reports > 10,000 rows — Workaround: use date filters
---
_Generated by SDD Release Notes Writer. Approved by: María García (Analyst)_
```

## Output
- `docs/releases/vX.Y.Z.md` — release notes in markdown
- Azure DevOps: comment on work items included in the release

## Constraints

- **NEVER fabricate work items.** Every entry MUST reference a real {prefix}ID from the project's work item system.
- **ALWAYS link to real {prefix}IDs.** Do NOT invent IDs or use placeholder numbers.
- **REQUIRE Analyst approval** before publishing. Include `Approved by: [name]` only after explicit confirmation.
- **NEVER include work items still in progress.** Only items with stateCategory = Completed qualify.
- **ALWAYS write descriptions in Spanish** to match the team's working language.
- **ALWAYS include breaking changes section** if any exist — never silently omit them.
- **ONE release note per version.** Do not combine multiple versions into a single document.

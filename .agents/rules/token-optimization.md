# Token Optimization

## Verbosity

Read `verbosity` from `.sdd-config.json` (default: `detailed`).

Both modes execute the FULL mandatory checklist internally. Verbosity controls ALL user-facing communication: questions, status, confirmations, and errors.

### `detailed` (default — recommended for adherence)

**Questions** — explain context, offer options with descriptions:
```
product.md existe pero está sin completar, y project_type está vacío.
No hay código en el proyecto, así que toca pasar por project-initializer.

¿Me cuentas sobre el proyecto?
- Nombre y propósito
- Audiencia
- Stack que tienes en mente (o lo defino después)
```

**Status** — show each step with icons:
```
✅ Work item AB#104336 creado en ADO (Requirement)
🌿 Branch: feature/AB#104336-registro-empleadores (creada desde dev)
📋 Requirements — 4 REQs + 2 diagramas
```

**Confirmations** — each result on its own line:
```
✅ Guardado en specs/AB#104336-registro-empleadores/requirements.md
✅ Approval: - [x] Analista: jsolano — 2026-07-01
✅ ADO: AB#104336 → Active
💾 Commit: docs(specs): create requirements...
✅ Pushed a feature/AB#104336-registro-empleadores
```

### `brief`

**Questions** — one line, no context dump:
```
Proyecto vacío. ¿De qué trata? (nombre, propósito, audiencia)
```

**Status** — compress to one line:
```
✅ AB#104336 · 🌿 feature/AB#104336-registro-empleadores
```

**Confirmations** — compress to one line:
```
✅ req · ✅ ADO · 💾 pushed · ¿Design?
```

**Rules for `brief`**:
- **Workflow** (status, confirmations, questions): MAX 2 lines per response
- **Content** (ideas, explanations, suggestions): Compact lists or tables. No section headers. Half the length of detailed.
- ONE question per turn, not multiple bullets
- No context explanation — assume user knows the flow
- Match user energy — "dame ideas rápidas" → bullet list, no paragraphs
- Tables are OK (same in both modes)

### ⚠️ Verbosity overrides power output format

Powers/skills say WHAT to communicate (Show, Ask, Inform, Confirm). This steering controls HOW:

| Power says | `detailed` | `brief` |
|---|---|---|
| `Show proposal` | Full block with context | 1-line summary + "¿Ajustes?" |
| `Ask once: "Adjust or proceed?"` | Full question with options | "¿OK?" |
| `Show summary and ask: "Commit?"` | Multi-line summary | "💾 ¿Commit?" |
| `Inform: WI created` | "✅ AB#123 creado en ADO (Requirement)" | "✅ AB#123" |
| `Inform: branch created` | "🌿 Branch: feature/AB#123-name (desde dev)" | "🌿 feature/AB#123-name" |
| `BLOCK: "Requirements necesita..."` | Full explanation + what to do | "⛔ Falta aprobación de requirements" |
| `Report error` | Full error + context + suggestion | "❌ [error] → [fix]" |

This applies to ALL powers, steerings, and skills. No exceptions.

## Communication style

- Be human, warm, conversational. Speak like a colleague.
- Execute first, explain after. Don't narrate steps before doing them.
- Propose defaults: "I'll use X unless you prefer Y." Don't ask what you can infer.
- Match the user's energy — short answers get short responses.
- After approval, execute. Don't re-confirm.
- Respond in the user's language. Technical notation may stay in English.
- Group related questions — one complete proposal, ask once.
- If user says "you decide" → decide immediately.

### Don't
- "Sure!", "Great question!", "I'd be happy to help"
- Narrate reasoning or internal state (config values, memory entities, frontmatter)
- Sound like a system log — "I don't have info yet" not "project_type is empty"
- Dump full documents in chat — summary only (max ~30 lines), save to file

## Artifact output

- **Chat**: Summary table/bullets (max ~30 lines). Never dump full document.
- **File**: Save complete content to `specs/{prefix}[ID]-[name]/artifact.md`
- **Tracker**: Update work item if `spec_prefix` configured
- **Templates**: Always use `specs/_templates/` as base. Never create from scratch.
- **Language**: Generated files (specs, ADO fields) use `docs_language` from `.sdd-config.json` (e.g., `"es"`, `"en"`). Chat follows user's language. Technical notation (EARS, Mermaid) always in English.
- After saving: `✅ Guardado → 📄 [artifact.md](file:///absolute/path)` + approval question + `📍 [what's next in lifecycle]`. User clicks → opens in editor → reviews.
- Before creating: check if artifact exists → update, don't recreate.

### Provenance headers

Every spec file (requirements.md, design.md, tasks.md, test-plan.md, migration-plan.md) MUST start with:

```
> 📋 Generated by `[power-name]` · [YYYY-MM-DD]
> ✅ Approved by: [pending | name (date)]
```

Update "Approved by" when user approves (`- [x]`).

product.md generated by scanner/initializer: `<!-- Generated by [power] on [date]. Safe to edit. -->`

.sdd-config.json: include `"_generated": { "by": "[power]", "date": "[YYYY-MM-DD]" }`

## Tool discipline

Use native IDE tools instead of terminal commands:

| Instead of | Use |
|---|---|
| `cat`, `head` | `view_file` with StartLine/EndLine |
| `grep -r` | `grep_search` |
| `ls`, `dir` | `list_dir` |
| `sed`, `echo >>` | `replace_file_content` / `write_to_file` |

Exceptions: git operations, build/test commands, piped data transforms.

## Sequential thinking

Use `sequential-thinking` MCP for complex decisions: architecture choices, debugging multi-cause issues, trade-off evaluation. Skip for simple tasks (rename, fix typo, single-file changes).

## Error handling

- Always report **exact error text** (stack trace, error code, message)
- Format: `❌ [what failed]: [exact error] → [next action]`
- Don't say "failed" without the actual error
- Don't retry silently more than once
- Don't inject env vars or modify configs to bypass errors
